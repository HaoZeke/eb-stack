replacement for those rules.

* Host: ~rg.surf~ (EasyBuild + residual)

Run ~eb~, ~eb-stack ingest~, ~eb --robot~ installs (*builds* claim), and any
**local-ai agent** residual session on **~rg.surf~** (SURF workstation). Do
not use the laptop as a fake EasyBuild host; do not use ~rg.terra~ for this
path (terra is the **cargo** builder for the ~eb-stack~ crate only).

Residual agents run **inside a herdr pane** on that host (see skill §7) —
not ad-hoc ~ssh rg.surf 'hermes …'~ / ~omp -p~.

#+begin_src bash
ssh rg.surf

If ~rg.surf~ is unreachable, stop and report.

* Claims: resolves ≠ builds ≠ binary-verified

| Rung | Proved by |
|------+-----------|
| *resolves* | ~eb-stack check-recipe~ (deps exist; packaging gate) |
| *builds* | ~eb --robot~ green; module file exists (**on ~rg.surf~**) |
| *binary-verified* | module load; binary runs / ~ldd~ / RPATH checks |

~eb-stack ingest~ only helps the bootstrap toward *resolves*. It does **not**
auto-author a landable PR end-to-end.

* Prefer search-and-tweak over inventing

#+begin_src bash
eb -S SoftwareName
#+end_src

If a nearby easyconfig exists, copy it and update version/toolchain per the
"Updating existing easyconfigs for a new toolchain" section of the writing
docs — that is EasyBuild's preferred path. Use ~eb-stack bump~ for
generation retargets of existing recipes.

* Optional bootstrap from foreign recipes

#+begin_src bash
ROBOT=/path/to/easybuild-easyconfigs/easybuild/easyconfigs

eb-stack ingest \
  --source path/to/recipe.yaml \   # or meta.yaml / package.py
  --toolchain-name foss \
  --toolchain-version 2024a \
  --easyconfigs "$ROBOT" \
  --keep-old-deps \
  --out-dir work/easyconfigs
#+end_src

Expect a *scaffold* with identity fields and (when the robot has candidates)
generation-native dependency versions. Expect residual warnings for product
flags, checksums, ~moduleclass~, and sanity paths. **Do not treat the scaffold
as a landable PR** or as a *builds* claim.

* Finish as a real easyconfig (required)

1. Prefer ~%(version)s~ / ~SOURCE_TAR_GZ~ / ~GITHUB_SOURCE~ templates.
2. Inject real SHA256 checksums:
   #+begin_src bash
   eb --inject-checksums work/easyconfigs/*/*/*.eb
   #+end_src
3. Set a *known* ~moduleclass~ (~eb --show-default-moduleclasses~).
4. Set ~sanity_check_paths~ to real install-relative files (default is only
   non-empty ~bin~ + ~lib~/~lib64~).
5. Contributor gate:
   #+begin_src bash
   eb --check-contrib work/easyconfigs/*/*/*.eb
   #+end_src
6. Dry-run and install with EasyBuild robot resolution:
   #+begin_src bash
   eb Name-Ver-foss-YYYY.eb -Dr --robot work/easyconfigs:$ROBOT
   eb Name-Ver-foss-YYYY.eb --robot work/easyconfigs:$ROBOT
   #+end_src

* Contribute upstream

Per https://docs.easybuild.io/contributing/ :

- Target the ~develop~ branch of ~easybuild-easyconfigs~.
- Prefer ~eb --new-pr~ / ~eb --update-pr~.
- Title pattern: ~{moduleclass}[toolchain] Name version~.
- Requirements: style (~--check-contrib~), consistency (~eb --review-pr~),
  test reports, maintainer approval.

* Mechanical first; local-ai agent only for judgment

On ~rg.surf~, run every mechanical step (~eb-stack ingest~ / MCP ~eb_ingest~,
~eb --inject-checksums~, ~eb --check-contrib~, ~eb-stack check-recipe~,
~eb -Dr --robot~) before involving a model.

Ingest always writes ~{stem}.residuals.json~ (or ~--residual-queue PATH~):
classified residual items + a claim ladder that does **not** claim
*resolves*/*builds*. Feed that JSON to the residual agent.

~check-recipe~ is hierarchy-aware: unpinned deps must match a generation
member (older GCCcore alone is not enough for a newer foss recipe).

Product flags, ambiguous ~moduleclass~, real ~sanity_check_paths~, companion
recipes for robot holes, and variant policy are *judgment residuals*: a
**local-ai agent** (Hermes preferred, in **herdr** on ~rg.surf~) follows
~skills/new-package/SKILL.md~ §7 — do **not** hardcode them into ~eb-stack~.
