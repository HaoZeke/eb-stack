#+OPTIONS: toc:nil num:nil

In this tutorial you take a real GROMACS easyconfig built against
~foss-2023b~ and bump it to the next toolchain generation, ~foss-2024a~,
with *zero hand-fed dependency versions*: every ~dependencies~ and
~builddependencies~ entry is resolved automatically from a directory of
easyconfigs, one command, no ~--dep~ flags.

.. figure:: /images/00-hero-annual-bump.jpg
   :alt: Mascot stamping foss-2024a onto a stack of easyconfig cards; residual sticky note for hand-added deps
   :align: center
   :width: 88%

   The mechanical majority of a generation bump is rewriting recipes and
   dependency versions; a short residual list stays human.
#+end_export

Everything else in the file -- ~easyblock~, ~homepage~, ~description~,
~toolchainopts~, ~source_urls~, ~sources~, ~patches~, ~checksums~,
~configopts~, ~exts_list~, ~moduleclass~ -- comes out identical to the
source.

* Install eb-stack

#+begin_src bash
git clone https://github.com/HaoZeke/eb-stack.git
cd eb-stack
cargo test
cargo build --release
# binary: target/release/eb-stack
#+end_src

The rest of this tutorial calls the binary as ~eb-stack~ (put
~target/release~ on your ~PATH~, or use the full path) and runs commands
from the repository root, since it uses easyconfigs already checked into
the repo's own test fixtures -- no files to create by hand.

* The source easyconfig

~tests/repro_fixtures/gromacs/GROMACS-2024.4-foss-2023b.eb~ is a real,
unmodified maintainer easyconfig -- 90 lines with the usual ~homepage~,
~description~, ~source_urls~, ~patches~, ~checksums~, and ~exts_list~
fields. The lines this tutorial cares about are the toolchain and the
dependency lists:

#+begin_src python
toolchain = {'name': 'foss', 'version': '2023b'}
toolchainopts = {'openmp': True, 'usempi': True}

builddependencies = [
    ('CMake', '3.27.6'),
    ('scikit-build-core', '0.9.3'),
]

dependencies = [
    ('Python', '3.11.5'),
    ('SciPy-bundle', '2023.11'),
    ('networkx', '3.2.1'),
    ('mpi4py', '3.1.5'),
]
#+end_src

* The dependency universe

~tests/repro_fixtures/universe_foss_2024a/~ holds real ~foss-2024a~-era
easyconfigs at every sub-toolchain level GROMACS's dependencies actually
live at: ~Python~ and ~CMake~ on ~GCCcore-13.3.0~, ~SciPy-bundle~ and
~networkx~ on ~gfbf-2024a~, ~mpi4py~ on ~gompi-2024a~. None of them sit on
plain ~foss-2024a~ -- that is the point: EasyBuild's composite toolchains
pull dependencies from several sub-toolchain levels of the *same
generation*, and ~eb-stack~ has to walk that hierarchy to find them.

* Bump it, zero hand-fed versions

#+begin_src bash
eb-stack bump \
  --source tests/repro_fixtures/gromacs/GROMACS-2024.4-foss-2023b.eb \
  --toolchain-name foss \
  --toolchain-version 2024a \
  --easyconfigs tests/repro_fixtures/universe_foss_2024a \
  --out-dir /tmp/gromacs-2024a
# wrote /tmp/gromacs-2024a/GROMACS-2024.4-foss-2024a.eb (from tests/repro_fixtures/gromacs/GROMACS-2024.4-foss-2023b.eb)
#+end_src

No ~--dep~ flags. ~--easyconfigs~ points at the universe directory above;
~eb-stack~ parses every easyconfig under it, walks the ~foss-2024a~
sub-toolchain hierarchy (~GCCcore~, ~GCC~, ~gfbf~, ~gompi~, ... down to
~SYSTEM~), and for each of ~CMake~, ~scikit-build-core~, ~Python~,
~SciPy-bundle~, ~networkx~, ~mpi4py~ picks the version that generation
actually used -- never older than the source's own version. No
~--version~ flag either, so the application version stays ~2024.4~: the
same release rebuilt against a newer toolchain, the common case for a
toolchain-generation bump.

* Inspect the result

#+begin_src bash
diff tests/repro_fixtures/gromacs/GROMACS-2024.4-foss-2023b.eb \
     /tmp/gromacs-2024a/GROMACS-2024.4-foss-2024a.eb
#+end_src

The diff is exactly the toolchain line and the six dependency versions
shown at the top of this page. Run the same command twice: both runs
produce the exact same bytes, because ~bump~ rewrites targeted spans of
the source text -- it does not reformat or regenerate anything else.

* What auto-resolve does not do

Compare the emitted file against
~tests/repro_fixtures/gromacs/GROMACS-2024.4-foss-2024a.eb~, the real
maintainer's ~foss-2024a~ recipe checked into the same fixture directory:
it also adds a new runtime dependency, ~('pybind11', '2.12.0')~, that does
not exist anywhere in the ~foss-2023b~ source. ~eb-stack~ correctly leaves
that line out -- a mechanical version bump cannot invent a dependency the
source recipe never declared. That one line is the only place the emitted
file and the real target differ, and the gap is intentional: see
[[file:explanation/fidelity.org][reproduction fidelity and limitations]]
for how often auto-resolve reproduces a real maintainer PR exactly, and
why the residual cases look the way they do.

* Override one dependency by hand

~--dep NAME=VERSION~ still works alongside ~--easyconfigs~: a hand override
wins over whatever auto-resolve would have picked for that name, and every
other dependency still auto-resolves normally.

#+begin_src bash
eb-stack bump \
  --source tests/repro_fixtures/gromacs/GROMACS-2024.4-foss-2023b.eb \
  --toolchain-name foss \
  --toolchain-version 2024a \
  --easyconfigs tests/repro_fixtures/universe_foss_2024a \
  --dep scikit-build-core=0.12.0 \
  --out-dir /tmp/gromacs-2024a-override
# scikit-build-core is 0.12.0 (the override), everything else auto-resolved as before
#+end_src
