#+TITLE: Architecture
#+OPTIONS: toc:t num:nil

~eb-stack~ is a pipeline of small, independently testable modules. Nothing
in the pipeline builds software or shells out to EasyBuild itself: the tool
reads easyconfig text, decides a consistent selection, and writes text
(lock, SBOM, reports, or a rewritten easyconfig) back out.

* Modules

- ~eb_parse~ (~src/eb_parse.rs~) :: Evaluates ~.eb~ files as a restricted
  Python-like DSL into ~ResolvedEasyconfig~ (~name~, ~version~,
  ~versionsuffix~, ~toolchain~, ~dependencies~, ~builddependencies~,
  ~exts_list~, templates expanded), then maps that to the solver-facing
  ~Candidate~ / ~DepReq~ shapes. Tree walks (~parse_easyconfig_tree~,
  ~parse_easyconfig_trees~) skip unparseable files individually instead of
  aborting, and merge multiple roots with later-path overlay precedence
  (~merge_candidates_with_precedence~). See
  [[file:parser-approach.org][the parser approach]].
- ~eb_template_constants~ (~src/eb_template_constants.rs~) :: The EasyBuild
  ~TEMPLATE_CONSTANTS~ table (~SOURCE_TAR_GZ~, ~GITHUB_SOURCE~,
  ~PYPI_SOURCE~, and around fifty others) that ~eb_parse~ seeds into every
  recipe's evaluation environment before resolving ~%(...)s~ templates.
- ~domain~ (~src/domain.rs~) :: The shared vocabulary every other module
  speaks: ~Toolchain~, ~Candidate~, ~DepReq~, ~Universe~, ~Policy~,
  ~StackLock~, ~LockPackage~. Plain serde structs, no behavior beyond a
  couple of small helpers (~Policy::effective_root_priority~).
- ~resolvo_provider~ (~src/resolvo_provider.rs~) :: ~EbProvider~ implements
  resolvo's ~DependencyProvider~ trait over the candidate set: which
  versions exist per package name, which versions satisfy a requirement
  range, and the dependency edges to expose to the SAT core. Also owns
  ~solve_with_resolvo~, the priority-lexicographic optimization loop layered
  on top of feasibility. See
  [[file:optimization-model.org][the optimization model]].
- ~select~ (~src/select.rs~) :: Thin wrapper that calls
  ~resolvo_provider::solve_with_resolvo~, turns the result into a
  ~StackLock~ (sorted, timestamped, tagged with the solver engine string),
  and validates every declared root is present.
- ~sbom~ (~src/sbom.rs~) :: ~StackLock~ (plus, when available, the
  universe's declared dependency edges) to a CycloneDX JSON document,
  written only when a caller asks for it (~--sbom-out~ is opt-in, no
  default filename). See [[file:../reference/formats.org][the format
  reference]].
- ~report~ (~src/report.rs~) :: ~StackLock~ to an install-ordered build list
  (topological sort) and a baseline-vs-solved markdown stack diff.
- ~hierarchy~ (~src/hierarchy.rs~) :: Sub-toolchain hierarchy membership and
  safe dependency-version resolution for ~bump --easyconfigs~: given a
  target toolchain generation (~foss-2024a~), which sub-toolchain levels
  (~GCCcore~, ~GCC~, ~gfbf~, ~gompi~, ..., ~SYSTEM~) belong to it, and, for
  each dependency name, the newest candidate in that hierarchy that is
  never older than the source recipe's own version and respects a
  ~versionsuffix~ pin. Ground truth for hierarchy membership is EasyBuild's
  own ~get_toolchain_hierarchy~, captured as fixtures under
  ~fixtures/toolchain_hierarchy/~.
- ~eb_emit~ (~src/eb_emit.rs~) :: The ~bump~ path. Rewrites the ~toolchain~
  dict, optional ~version~ assignment, named dependency tuples, and (on a
  version bump) the source checksum entry in an easyconfig's *source text*,
  byte-preserving everything else. When ~--easyconfigs~ is given, it calls
  ~eb_parse~'s tree walk and ~hierarchy~'s resolver to fill dependency
  versions before rewriting -- but it still never builds a ~Universe~ or
  calls the SAT solver: auto-resolve is dependency-version lookup, not
  co-selection.
- ~version~ (~src/version.rs~) :: EasyBuild-shaped version comparison and
  the ~version_req~ matcher (~==~, ~>=~, ~<=~, ~>~, ~<~, bare-exact) shared
  by the parser, the provider, the hierarchy resolver, and the
  baseline-generation picker.

* Two entry paths, one codebase

~solve~ / ~solve-json~ / ~parse~ / ~sbom~ all flow through the co-selection
pipeline above: parse (or load an already-built universe JSON), build a ~Universe~, ask
~select_stack~ for a consistent ~StackLock~, then derive reports from that
lock.

~bump~ never touches the solver, with or without ~--easyconfigs~: it reads
one file, rewrites specific spans of its text, and writes the result. The
auto-resolve universe (when ~--easyconfigs~ is given) reuses ~eb_parse~'s
tree walk to build a set of candidates and ~hierarchy~ to pick a version per
dependency name -- a lookup, not a co-selection -- so ~bump~ still has no
dependency on a policy or on resolvo. That separation is why bumping one
recipe never requires every other recipe in the tree to be jointly
consistent: it is a single-recipe operation, informed by but not
constrained by the rest of the universe.

* Where the co-selection graph's edges come from

Every ~Candidate~ carries two separate dependency lists: ~dependencies~
(runtime) and ~builddependencies~ (build-time only, e.g. ~CMake~). Both
lists become co-selection *requirements* on the SAT core identically
(~resolvo_provider::get_dependencies~ chains them) -- a package with an edge
to another only through ~builddependencies~ still forces that other package
into the lock. The role distinction survives only in the outputs that care
about it: the SBOM's runtime-vs-build-time ~dependsOn~ maps
(~dep_map_from_universe~ vs ~build_dep_map_from_universe~ in ~sbom.rs~) keep
the two separate even though the solver does not.
